In the two previous issues of this Newsletter, we talked about the [Use Case pattern](/newsletter/frontend-architecture-use-case-pattern) and the [Repository Pattern](/newsletter/why-you-should-use-the-repository-pattern-to-interact-with-data) and in this newsletter, you'll see the true potential unlocked by following said patterns.

When we follow the use case pattern we are actually making use of the [Command Design Pattern](https://refactoring.guru/design-patterns/command), since all use cases are devised to be executed using the same method: `handle()`.

```typescript
export class GetDestinationsQry implements UseCase<void, Destination[]> {
  constructor(
    private readonly destinationRepository: DestinationRepository,
    private readonly destinationOrderer: DestinationOrderer,
  ) {}

  async handle(): Promise<Destination[]> {
    return this.destinationOrderer.order(
      await this.destinationRepository.findAll()
    )
  }
}
```

Here is another use case:

```typescript
export class CreateDestinationCmd implements UseCase<CreateDestination, void> {
  constructor(private readonly destinationRepository: DestinationRepository) {}

  async handle(createDestination: CreateDestination): Promise<void> {
    return this.destinationRepository.create(createDestination)
  }
}
```

**A use case must implement the `handle()` method to be considered a Use Case**.

## Why?

So, why would we want to execute use cases in the same way? Well, because then we can **add logic before, after or during the execution of use cases** in a simple yet powerful way.

Let me introduce you the `UseCaseService`:

```typescript
export class UseCaseService {
  execute<In, Out>(useCase: UseCase<In, Out>, params?: In): Promise<Out> {
    return useCase.handle(params)
  }
}
```

It's meant to be used like this:

```typescript
export const destinationInMemoryRepository = new DestinationInMemoryRepository()
export const createDestinationCmd = new CreateDestinationCmd(destinationInMemoryRepository)
export const useCaseService = new UseCaseService()

const destinations = await useCaseService.execute(createDestinationCmd, newDestination)
```

Ok, so what? All of this code _just_ to add another abstraction? What if we needed to implement logging for every use case? **Easy**.

```typescript
export class UseCaseService {
  execute<In, Out>(useCase: UseCase<In, Out>, params?: In): Promise<Out> {
    console.log('Logging use case:', useCase.constructor.name)
    console.log('Logging params:', params)
    return useCase.handle(params)
  }
}
```

Now every use case has logging capabilities. Want also to log the result? We can await the `useCase.handle(params)`:

```typescript
export class UseCaseService {
  async execute<In, Out>(useCase: UseCase<In, Out>, params?: In): Promise<Out> {
    console.log('Logging use case:', useCase.constructor.name)
    console.log('Logging params:', params)
    const result = await useCase.handle(params)
    console.log('Logging result:', result)
    return result
  }
}
```

What if we need a **global error handling system** to display the user a friendly error message?

```typescript
export class UseCaseService {
  async execute<In, Out>(useCase: UseCase<In, Out>, params?: In): Promise<Out> {
    try {
      console.log('Logging use case:', useCase.constructor.name)
      console.log('Logging params:', params)
      const result = await useCase.handle(params)
      console.log('Logging result:', result)
      return result
    } catch (e) {
      alert(e.message)
      throw e
    }
  }
}
```

> ![NOTE]
> We will improve the `UseCaseService` so we use the [Chain of Responsibility Pattern](https://refactoring.guru/design-patterns/chain-of-responsibility) to improve how we handle cross-cutting concerns in a way that is maintainable.

With this approach, **you don't need to keep adding try catch all over your code, you can handle them in just one place**. If you still need to handle a particular error, you can still do so, but by default, the system is prepared to handle them generally.

## Conclussion

The `UseCaseService` is a wonderful place to handle transversal core features. Over the years I've used it to solve really complex problems in an easy way. Here are some of the use cases of the `UseCaseService` that I've applied:

* Caching
* Logging
* Offline capabilities
* Confirmation message
* Success messages
* Retries
* Timers
* Roles and permissions handling
* Loading strategies
* Many more
